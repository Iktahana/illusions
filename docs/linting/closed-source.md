# クローズドソース・ルールセット配布

ルールセットは**コードモジュール**なので、ソースを隠して配布できる。ただし保護強度には限界がある。

## 段階的な保護強度

| 段階             | 方法                                                                         | 強度       | 備考                                         |
| ---------------- | ---------------------------------------------------------------------------- | ---------- | -------------------------------------------- |
| ① 素の JS        | ビルド済み `index.js` を配布（`src/` 非公開）                                | 弱         | ソース非公開なだけ                           |
| ② 難読化 JS      | terser / javascript-obfuscator で minify+難読化                              | 中（抑止） | worker が Blob import するだけなので動作する |
| ③ WASM           | Rust / AssemblyScript → `.wasm`。薄い JS ラッパで `createRules` 契約を満たす | 強（抑止） | 解析難度は上がる                             |
| ④ サーバー側評価 | ロジックを端末に降ろさず API で評価（dict REST / L3 LLM と同方式）           | 真の関源   | オフライン不可・コスト                       |

## 正直な限界

- ①〜③は**クライアント実行**である限り抑止であって完全保護ではない。最終的に端末上で実行されるため、
  実行時にメモリ/モジュールを捕捉でき、暗号化しても**復号鍵がアプリに載る**以上は突破され得る。
- **真にソースを守る**には ④（サーバー側評価）しかない。常時接続が前提となるルールに限定して使う。

## 単一ファイル配布（`.illruleset`）

1ファイルで配布したい場合は `.illruleset` コンテナを使う（実装は配布フェーズ）。

- 構造: **平文ヘッダ(JSON: magic / containerVersion / manifest / payload 情報) ＋ ペイロード(②難読化JS or ③WASM)**。
- `manifest` を平文に保つ理由は不変（コード非実行で読む）。隠せるのはペイロードのみ。
- 型は `lib/linting/registry/ruleset-source.ts` の `IllrulesetContainerHeader` に予約済み。

## 既存ライセンス概念との整合

`GUIDELINES` には既に `Paid` ライセンスのガイドライン（記者ハンドブック・JTCA 等）がある。
クローズドな有料ルールセットはこの概念と整合する。マーケット/署名/ライセンス検証は配布フェーズで設計する。
